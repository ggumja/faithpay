# [상세 PRD v3.0] FaithPay: Multi-tenant Religious Offering SaaS

* **제품명:** FaithPay (페이쓰페이)
* **대상:** 온라인 결제 도입이 필요한 모든 종교 단체 (개신교, 불교, 천주교 등)
* **핵심 가치:** * **Zero-Config:** 가입 즉시 종교별 맞춤 템플릿 제공.
    * **Trust & Transparent:** 투명한 헌금 관리 및 실시간 정산 리포트.
    * **Devotion UX:** 종교적 감수성을 유지하는 결제 흐름.

## 3. 핵심 기능 요구사항 (Functional Requirements)

### 3.1. 멀티테넌시 테마 엔진 (Religion-Specific Theme Engine)
* **[FR-101] 동적 UI 렌더링:** `TenantID`에 따라 CSS 변수(Primary Color), 폰트, 아이콘 세트를 동적으로 로드.
* **[FR-102] 용어 치환 시스템(I18n 확장):** * 기독교: "헌금하기", "기도제목", "성도님"
    * 불교: "보시하기", "발원문/사주", "불자님"
    * 천주교: "봉헌하기", "미사지향", "교우님"
* **[FR-103] 커스텀 위젯:** 관리자가 예배 영상(YouTube), 공지사항, 이번 주 성경 구절 등을 결제 메인에 배치 가능.

### 3.2. 봉헌 및 결제 시스템 (Payment Engine)
* **[FR-201] 하이브리드 결제 모듈:**
    * **단발성 결제:** 신용카드, 간편결제(Toss, Kakao, Apple, Samsung).
    * **정기 결제(Billing):** 십일조/교무금 등을 위한 자동 결제(Schedules) 등록 및 관리.
    * **가상계좌 발급:** 고령층을 위해 입금 시 즉시 알림이 가는 전용 가상계좌 제공.
* **[FR-202] 종교 특화 입력 필드:**
    * **불교:** 가족 구성원별 사주(간지) 입력란 및 연등 번호 자동 배정 로직.
    * **천주교:** 영명(세례명) 검색 및 미사 지향(생미사/연미사) 구분 선택.
* **[FR-203] 오프라인 QR 연동:** 성전/법당 좌석에 부착된 QR 스캔 시 해당 항목 결제창으로 즉시 연결.

### 3.3. 관리 시스템 및 정산 (Admin & Settlement)
* **[FR-301] 실시간 정산 대시보드:** * `총 봉헌액 - (PG 수수료 + SaaS 이용료) = 입금 예정액` 실시간 표기.
* **[FR-302] 기도문/발원문 워크플로우:**
    * 신도가 작성한 내용을 관리자가 '확인 완료' 처리 시 신도에게 "기도 수렴 완료" 알림 발송.
    * 대량 인쇄 기능을 통해 법당/제단 비치용 명단 PDF 자동 생성.
* **[FR-303] 기부금 영수증 자동화:** * 연말정산 시기에 맞춰 국세청 연동용 자료(CSV/API) 추출 및 개별 PDF 발급.

## 4\. 데이터 엔티티 설계 (Core Entities)

### `Tenant` (단체)

  * `id` (UUID): 고유 식별자
  * `name` (String): 단체명
  * `religion_type` (Enum): CHRISTIAN, BUDDHISM, CATHOLIC
  * `theme_config` (JSON): 컬러, 폰트, 이미지 등 UI 설정값
  * `payout_account` (Encrypted): 정산 계좌 정보

### `Donation` (결제 내역)

  * `id` (UUID): 거래 고유 번호
  * `tenant_id` (FK): 해당 단체
  * `member_id` (FK): 결제 신도 (비회원 시 null)
  * `amount` (Integer): 금액
  * `item_type` (String): 헌금/보시 종류
  * `metadata` (JSON): 기도제목, 사주, 지향 등 종교별 특화 데이터
  * `status` (Enum): PENDING, COMPLETED, CANCELLED

## [기술 스택 제안서] FaithPay SaaS Platform

### 1. Frontend: 사용자 경험 및 테마 커스텀 최적화
단체별로 다른 도메인과 디자인을 제공해야 하므로, SSR(Server Side Rendering) 성능이 뛰어난 프레임워크가 필수적입니다.

* **Framework:** **Next.js (App Router)**
    * **이유:** 각 종교 단체별 고유 URL(SEO) 처리와 서버 컴포넌트를 이용한 빠른 초기 로딩 속도 보장.
* **Styling:** **Tailwind CSS + Shadcn UI**
    * **이유:** `tailwind.config.js`와 CSS 변수를 활용해 종교별 테마(컬러, 폰트)를 런타임에 동적으로 변경하기 매우 용이함.
* **State Management:** **TanStack Query (React Query)**
    * **이유:** 실시간 결제 상태 확인 및 관리자 대시보드의 데이터 동기화 최적화.

---

### 2. Backend: 멀티테넌시 및 확장성 중심
수많은 종교 단체의 데이터를 안전하게 격리하고, 복잡한 비즈니스 로직을 처리해야 합니다.

* **Language & Framework:** **Node.js (NestJS)**
    * **이유:** 모듈형 구조로 설계되어 '기독교 모듈', '불교 모듈' 등을 독립적으로 개발하기 좋으며, TypeScript 지원으로 대규모 정산 로직의 안정성 확보.
* **Database:** **PostgreSQL**
    * **이유:** 복잡한 관계형 데이터(정산, 교적) 처리에 강력하며, **RLS(Row Level Security)** 기능을 통해 DB 레벨에서 테넌트 간 데이터 격리를 완벽하게 구현 가능.
* **ORM:** **Prisma** or **Drizzle ORM**
    * **이유:** 타입 안정성이 뛰어나며, 멀티테넌시 쿼리 필터를 적용하기에 유연함.

### 3. Infrastructure & DevOps: 주말 트래픽 대응 전략
평일 대비 주말 트래픽이 10배 이상 튈 수 있는 서비스 특성을 반영합니다.

* **Cloud:** **AWS (Amazon Web Services)**
* **Compute:** **AWS Fargate (ECS)** 또는 **App Runner**
    * **이유:** 서버리스 컨테이너 환경으로, 주말 오전 트래픽 급증 시 자동으로 인스턴스를 확장(Auto-scaling)하고 평일에는 비용을 절감.
* **Cache:** **Redis**
    * **이유:** 결제 중복 방지(Idempotency), 실시간 접속자 수 체크 및 세션 관리.
* **Storage:** **AWS S3** (고유번호증, 로고, 기부금 영수증 PDF 저장).
